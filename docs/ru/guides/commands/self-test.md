## Обзор 🎯
Запускает комплексные тесты самодиагностики для проверки установки, конфигурации и работоспособности gotr.

> [!TIP]
> Для быстрого старта по `self-test`: проверьте `--help`, затем выполните
> безопасный/проверочный запуск перед боевым сценарием.

## Синтаксис 🧩
```bash
gotr self-test [flags]
```

## Флаги ⚙️

```text
--failures-only     Показывать только неудачные проверки
-h, --help          справка для self-test
--include-skipped   Включить пропущенные проверки в вывод
--json              Вывести результаты в формате JSON
```

## Глобальные флаги 🌐

```text
-k, --api-key string    API ключ TestRail
-c, --config            Создать дефолтный файл конфигурации
-f, --format string     Формат вывода: table, json, csv, md, html (default "table")
--insecure              Пропустить проверку TLS сертификата
--non-interactive       Отключить интерактивные подсказки; завершить с ошибкой если требуется ввод
-q, --quiet             Подавить служебный вывод (прогресс, статистику, сообщения о сохранении)
--url string            Базовый URL TestRail
-u, --username string   Email пользователя TestRail
```

- Когда вы работаете именно с доменом `self-test` и хотите предсказуемый, повторяемый сценарий.
- Когда нужно быстро перейти от проверки параметров к рабочему запуску без лишних ручных шагов.

## 🧾 Ожидаемый результат выполнения

### Что считается успешным результатом

- Все критичные проверки проходят, а команда завершается с кодом `0`.
- В режиме `--failures-only` отсутствуют провальные пункты.

### Пример структуры вывода (`--json`)

```json
{
  "status": "ok",
  "checks": [
    { "name": "...", "status": "passed" }
  ]
}
```

Важно: имена и количество проверок зависят от среды и текущего состояния проекта.

```bash
gotr self-test --failures-only
```

## Практические советы 🧭

- ✅ **Совет: Фиксируйте шаблонные команды**
  > Храните типовые команды для проекта/сьюта/кейса в отдельном internal runbook, чтобы сократить ручные ошибки при повторных запусках.
  >
  > ---

- ✅ **Совет: Логируйте контекст запуска**
  > Перед выполнением сохраняйте ключевые параметры: идентификаторы, URL, выбранные флаги и время запуска. Это ускоряет разбор инцидентов.
  >
  > ---

- ✅ **Совет: Разделяйте диагностический и рабочий режим**
  > Сначала проверка (`--help`, безопасные флаги), затем боевой запуск. Такой режим уменьшает риск некорректных изменений в TestRail.

---

## Частые ошибки и диагностика 🛠️

- ⚠️ **Ошибка: Команда запускается, но результат «не тот»**
  > Проверьте корректность целевых ID и конкретную подкоманду; чаще всего проблема в несоответствии endpoint и переданных аргументов.
  >
  > ---

- ⚠️ **Ошибка: Неочевидный отказ в автоматизации**
  > Убедитесь, что в CI переданы все обязательные параметры и отключены интерактивные ожидания ввода там, где это необходимо.
  >
  > ---

- ⚠️ **Ошибка: Сложно сравнить результаты между запусками**
  > Сохраняйте выходные артефакты в файл и используйте единый формат вывода для повторяемого анализа.
