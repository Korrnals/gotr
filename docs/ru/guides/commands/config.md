## Обзор 🎯
Команды для работы с конфигурационным файлом утилиты gotr.

> [!TIP]
> Для быстрого старта по `config`: проверьте `--help`, затем выполните
> безопасный/проверочный запуск перед боевым сценарием.

## Синтаксис 🧩
```bash
gotr config [command]
```

## Подкоманды

| Подкоманда | Описание |
| --- | --- |
| `edit` | Открыть конфиг-файл в редакторе по умолчанию |
| `init` | Создать дефолтный файл конфигурации |
| `path` | Показать путь к текущему конфиг-файлу |
| `view` | Показать содержимое текущего конфиг-файла (чувствительные ключи маскируются) |

## Флаги ⚙️

```text
-h, --help   справка для config
```

## Глобальные флаги 🌐

```text
-k, --api-key string    API ключ TestRail
-c, --config            Создать дефолтный файл конфигурации
-f, --format string     Формат вывода: table, json, csv, md, html (default "table")
--insecure              Пропустить проверку TLS сертификата
--non-interactive       Отключить интерактивные подсказки; завершить с ошибкой если требуется ввод
-q, --quiet             Подавить служебный вывод (прогресс, статистику, сообщения о сохранении)
--url string            Базовый URL TestRail
-u, --username string   Email пользователя TestRail
```

## Примеры 🚀

### ▶️ Сценарий 1: Разведка возможностей команды
🎯 **Цель:** быстро понять доступные параметры и ограничения без выполнения рискованных операций.

```bash
gotr config --help
```

✅ **Что это даёт:** вы видите актуальный синтаксис, флаги и возможные ветки выполнения именно для текущей версии CLI.

---

### ▶️ Сценарий 2: Точечный запуск по целевому действию
🎯 **Цель:** проверить логику конкретной операции перед интеграцией в рабочий поток.

```bash
gotr config edit --help
```

✅ **Что это даёт:** вы валидируете правильную точку входа (подкоманду или форму вызова) и избегаете неверного выбора endpoint.

---

### ▶️ Сценарий 3: Безопасная проверка/структурированный вывод
🎯 **Цель:** получить предсказуемый результат для анализа или CI.

```bash
gotr config
```

✅ **Что это даёт:** безопасный или машинно-читаемый прогон для предварительной проверки результата.

---

### ▶️ Сценарий 4: Рабочий мини-пайплайн
🎯 **Цель:** показать практический поток «проверка → запуск → сохранение артефакта».

```bash
gotr config edit --help && gotr config edit
```

✅ **Что это даёт:** воспроизводимый шаблон, который легко перенести в скрипты и командные runbook.

---

## ⚡ Быстрый старт (30 секунд)

1. Быстро проверьте синтаксис и доступные флаги:
```bash
gotr config --help
```
2. Выберите рабочую ветку выполнения:
```bash
gotr config edit --help
```
3. Выполните проверочный/рабочий запуск:
```bash
gotr config edit --help
```

---

## 🧪 Чек-лист перед запуском

- [ ] Проверены `URL`, учетные данные и доступ к TestRail.
- [ ] Подтверждены корректные идентификаторы проекта/сьюта/кейса.
- [ ] Прогнан безопасный/диагностический запуск (`--help`, `--dry-run`, `--json` или `--save`).
- [ ] Зафиксирован целевой формат вывода и место сохранения артефактов.

---

## 🎯 Когда использовать

- Когда вы работаете именно с доменом `config` и хотите предсказуемый, повторяемый сценарий.
- Когда нужно быстро перейти от проверки параметров к рабочему запуску без лишних ручных шагов.

---

## 🚫 Когда не использовать

- Когда вы не уверены в целевых ID или endpoint: сначала уточните через `--help` и проверочный запуск.
- Когда нужна пакетная операция другого домена: выберите более специализированную команду/подкоманду.

---

## FAQ ❓

- ❓ **Вопрос:** Когда лучше использовать команду `config`?
  > ↪️ **Ответ:** когда задача относится к домену этой команды и вы хотите сохранить предсказуемый сценарий работы без смешивания с другими ресурсами.
  >
  > ---

- ❓ **Вопрос:** С чего начинать, если параметры неизвестны?
  > ↪️ **Ответ:** сначала запускайте `gotr config --help`, затем уточняйте справку у подкоманды. Это снижает риск неверной формы вызова.
  >
  > ---

- ❓ **Вопрос:** Какие подкоманды/ветки стоит проверять в первую очередь?
  > ↪️ **Ответ:** базовый набор для старта: edit, init, path, view. Начинайте с самого частого сценария в вашем процессе релизов/регресса.
  >
  > ---

- ❓ **Вопрос:** Как безопасно запускать в прод-контуре?
  > ↪️ **Ответ:** используйте пошаговую схему: проверка синтаксиса, ограниченный пробный запуск, затем финальный запуск с сохранением результата.
  >
  > ---

- ❓ **Вопрос:** Как встроить команду в CI/CD?
  > ↪️ **Ответ:** используйте стабильно формируемые параметры, структурированный вывод (если доступен) и явный контроль exit-code в пайплайне.

## 🧾 Ожидаемый результат выполнения

### Что считается успешным результатом

- Команды `view/path/edit/init` выполняются без ошибок и отражают актуальный конфиг.
- После `init` появляется рабочий конфигурационный файл с ожидаемыми ключами и правами доступа 0600.
- В `view` чувствительные ключи (`api_key`, `password`, `token`, `authorization`) выводятся как `"***"`.


---

## 🔎 Проверка результата через соседние команды

- После изменений конфига выполните любую read-команду (`get/list`) как smoke-check.

```bash
gotr config view
gotr list <resource>
```

## Практические советы 🧭

- ✅ **Совет: Фиксируйте шаблонные команды**
  > Храните типовые команды для проекта/сьюта/кейса в отдельном internal runbook, чтобы сократить ручные ошибки при повторных запусках.
  >
  > ---

- ✅ **Совет: Логируйте контекст запуска**
  > Перед выполнением сохраняйте ключевые параметры: идентификаторы, URL, выбранные флаги и время запуска. Это ускоряет разбор инцидентов.
  >
  > ---

- ✅ **Совет: Разделяйте диагностический и рабочий режим**
  > Сначала проверка (`--help`, безопасные флаги), затем боевой запуск. Такой режим уменьшает риск некорректных изменений в TestRail.

---

## Частые ошибки и диагностика 🛠️

- ⚠️ **Ошибка: Команда запускается, но результат «не тот»**
  > Проверьте корректность целевых ID и конкретную подкоманду; чаще всего проблема в несоответствии endpoint и переданных аргументов.
  >
  > ---

- ⚠️ **Ошибка: Неочевидный отказ в автоматизации**
  > Убедитесь, что в CI переданы все обязательные параметры и отключены интерактивные ожидания ввода там, где это необходимо.
  >
  > ---

- ⚠️ **Ошибка: Сложно сравнить результаты между запусками**
  > Сохраняйте выходные артефакты в файл и используйте единый формат вывода для повторяемого анализа.
